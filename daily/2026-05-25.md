# Daily Note: 2026-05-25

## Topic

Long-term Memory for AI Agents.

## Summary

Today's session reframed agent memory as an infrastructure problem rather than a database feature. The key distinction is that a database aims to store accurately and retrieve reliably, while agent memory must decide what to remember, when to recall, when to update, and when to forget.

## Key Ideas

- Agent memory is not CRUD. It requires selective forgetting, relevance-based recall, and the ability to overwrite stale state with newer evidence.
- Memory can be understood as a stack: embeddings, vector database, RAG, context engineering, and memory engineering.
- Content memory and time horizon should be separated:
  - semantic memory: facts;
  - episodic memory: events with context;
  - procedural memory: learned workflows and how-to knowledge;
  - working memory: temporary session state;
  - long-term memory: persistent cross-session state.
- The ecosystem is converging on memory as an agent infrastructure layer, but general-purpose memory remains difficult.

## My Takeaway

Memory is moving from feature to infrastructure. Once memory becomes mission-critical, it needs explicit design principles rather than ad hoc storage. This creates room for vertical memory layers, especially in domains with long timelines, audit requirements, and high consequences.

## Programmable Compliance Connection

For regulated or cross-border workflows, agent memory cannot simply remember everything. It needs boundaries:

- what should be persisted;
- what should expire;
- what can be recalled for future decisions;
- what needs human review before becoming durable memory;
- how old assumptions should be corrected when new evidence appears.

This is relevant to Programmable Compliance because compliance memory must be selective, reviewable, and accountable.
