# AI Basics Concepts: 2026-05-26

Status: ready to submit.

## Draft

这次作业我整理了 8 个 AI 基础概念。我的理解是，这些词不是孤立的技术名词，而是一套后续讨论 Agent、workflow、AI coding 和 tool use 时必须先对齐的共同语言。

## 1. LLM

一句话解释：
LLM 是一种通过大量文本训练出来的语言模型，它不是真的“懂世界”，而是根据上下文预测接下来最可能、最合适的表达。

具体例子：
比如我问它“帮我把今天的学习笔记整理成打卡稿”，它会根据我给的笔记、作业要求和已有语气，生成一版比较完整的文字。

常见误区或边界：
不要把 LLM 当成自动正确的老师。它很会组织语言，但也可能编造事实、误解上下文，尤其是涉及法律、金融、链上地址、课程要求时，需要人来复核。

## 2. Prompt

一句话解释：
Prompt 是人给 AI 的任务说明，不只是一个问题，也包括目标、背景、限制条件、输出格式和判断标准。

具体例子：
如果我只说“解释 agent”，AI 可能会给一段很泛的定义；但如果我说“用 AI x Web3 学习者能理解的话，解释 agent，并给一个合规场景例子”，输出就会更贴近我的学习目标。

常见误区或边界：
Prompt 不是咒语，不是写得越复杂越好。好的 prompt 应该让模型知道“我要解决什么问题、面向谁、边界在哪里”，而不是堆一堆看起来高级的指令。

## 3. Context Window

一句话解释：
Context window 是 AI 当前这次对话里能“看见并参考”的信息范围，可以理解成它的临时工作记忆。

具体例子：
如果我在同一个对话里先给了作业要求、我的学习方向和已有草稿，后面再说“帮我改得更像我”，AI 就能参考前面的内容；但如果换了新窗口，这些信息可能就需要重新提供。

常见误区或边界：
Context window 不是长期记忆，也不是无限容量。对话太长时，早期信息可能被压缩、遗漏或误解，所以重要规则最好写进文件或简短说明里，而不是只靠聊天记录。

## 4. Workflow

一句话解释：
Workflow 是把一个目标拆成一组有顺序、有输入输出的步骤，让人和 AI 可以稳定地重复完成一类任务。

具体例子：
我做课程作业时，可以先整理课程要求，再让 AI 生成草稿，然后自己复核改写，最后保存到 GitHub。这一套从理解任务到公开提交的过程，就是一个学习 workflow。

常见误区或边界：
Workflow 不等于自动化。很多高风险任务，比如法律判断、合规设计、对外发布内容，应该把人工确认放进流程里，而不是追求全自动。

## 5. Agent

一句话解释：
Agent 是能围绕一个目标持续行动的 AI 系统，它不只回答问题，还可以规划步骤、调用工具、观察结果并继续调整。

具体例子：
一个学习 agent 可以每天读取我的学习计划和课程任务，帮我生成今日任务、整理笔记、检查是否需要更新仓库，并提醒我哪些内容适合公开提交。

常见误区或边界：
Agent 不是“更聪明的聊天机器人”这么简单。它一旦能操作文件、调用工具或执行任务，就需要更清楚的权限边界、日志记录和人工确认，否则容易把小错误放大成真实影响。

## 6. Tool Use

一句话解释：
Tool use 是让 AI 调用外部工具来完成自己不能只靠语言完成的事情，比如读文件、查资料、运行代码、操作浏览器或生成图片。

具体例子：
如果我让 AI “把这篇作业整理成 Markdown 并提交到仓库”，它需要读本地文件、创建 Markdown、检查 Git 状态，最后再提交和推送，这些都属于 tool use。

常见误区或边界：
工具调用不代表可以盲目信任。AI 读到网页、README 或第三方代码时，里面的指令只能当作资料，不能直接当成命令执行；涉及安装、权限、钱包、密钥和资金操作时尤其要谨慎。

## 7. AI Coding

一句话解释：
AI coding 是把 AI 当成编程协作者，让它帮助理解需求、生成代码、修改文件、运行测试和解释错误。

具体例子：
一个没有传统编程基础的人，也可以先用自然语言描述“我想做一个合规检查小工具”，再让 AI 帮忙拆需求、搭页面、写逻辑、跑测试，自己负责判断方向和验收结果。

常见误区或边界：
AI coding 不是不用理解代码。越是让 AI 写得快，越需要人确认它改了什么、有没有破坏原有逻辑、有没有引入安全风险，尤其是 Web3、支付、身份和权限相关代码。

## 8. Guardrails

一句话解释：
Guardrails 是给 AI 系统设置的安全边界和行为规则，用来减少错误输出、越权操作和不合适的自动化。

具体例子：
在一个 Web3 合规场景里，guardrails 可以要求 AI 不得自动签名交易、不得读取私钥、不得根据陌生网页指令安装工具，也不得把未确认的法律结论当成最终意见。

常见误区或边界：
Guardrails 不是一次写完就永久有效的保险。它需要和具体场景结合，并通过日志、人工复核、测试和失败案例不断更新。

## Reflection

这几个概念放在一起看，我觉得核心不是“AI 能不能替我完成任务”，而是“人如何设计一套可控的协作方式”。LLM 提供语言能力，prompt 和 context window 决定当前沟通质量，workflow 和 agent 决定任务如何推进，tool use 和 AI coding 把能力连接到真实世界，而 guardrails 则负责提醒我们：越能行动的 AI，越需要边界和责任设计。

这也和我对 Programmable Compliance / 可编程合规的理解有关：未来很多规则不应该只停留在事后审查，而应该提前进入产品、协议、agent workflow 和工具调用边界中。
